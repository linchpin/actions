# linchpin/actions ref audit — 2026-06-12

Which linchpin-owned repos reference `linchpin/actions@<ref>` in their
workflows (GitHub code search across the org, default branches). Generated
while preparing v4 — use it to plan the `@main`/`@v2` cleanup and the
eventual `@v3` → `@v4` migration.

Summary: **62 repos / 252 references — 94 on `@main`, 86 on `@v3`, 72 on `@v2`.**
None of the repos are archived.

Key patterns:

- `@main` resolves to the **v2-era code** (main never received v3), so the
  `@main` and `@v2` groups run the same generation of workflows.
- The most common `@main` stragglers are `release-please.yml` (20 repos) and
  the WP Engine deploy workflows (34 files across prod/staging/develop).
- **Template sources spread the problem**: the org `.github` repo
  (workflow-templates), `deploy-scaffold`, and the `hari` /
  `rchagen-shopify-elite` generators all emit `@main` references into new
  projects. Fix these first to stop the bleeding.

## Suggested cleanup order

1. **Template sources** (`.github` workflow-templates, `deploy-scaffold`,
   `hari`, `rchagen-shopify-elite`, `autodeploy-scaffold-pantheon`) — emit
   `@v4` (or `@v3.0.0` until v4 GA) so new projects start clean.
2. **Active WP Engine clients on `@main`/`@v2`** — these get the most value
   from v4 (working caches, health checks); migrate per
   [MIGRATION-v3-to-v4.md](MIGRATION-v3-to-v4.md) once the WP Engine
   composite is validated on one site.
3. **`@v3` repos** — already on the current line; migrate opportunistically.
4. Re-run this audit after each wave (the generating command is in the doc
   history / can be re-run via gh code search for `org:linchpin "linchpin/actions"`).

> Caveats: code search covers default branches only, and this audit tracks
> `linchpin/actions` references — other linchpin-owned actions pinned to
> moving refs (e.g. `linchpin/actions-wpengine-ssh@main`,
> `linchpin/send-to-mantle@main` in v2-era workflows) ride along when those
> workflows migrate.
